# 🔒 MUUDAY-APP — Relatório de Segurança Completo

---

## 🔴 CRITICAL (3)

### 1. Escalação de privilégio via signup metadata
O trigger `handle_new_user()` lê o `role` direto do `raw_user_meta_data`. Um atacante pode chamar a API do Supabase diretamente com `role: 'admin'` e se tornar admin instantaneamente.

**Fix:** Trigger deve ignorar o role do metadata e sempre usar `'usuario'`. Promoção só via server-side.

### 2. API keys reais em arquivo rastreável
`.env.vercel-check` contém Supabase anon key + Vercel OIDC JWT token e **não está no `.gitignore`**. Se já foi pushado, as credenciais estão expostas.

**Fix:** Adicionar ao `.gitignore`, rotacionar todas as chaves expostas.

### 3. Duas RESEND_API_KEY no `.env.local`
A segunda sobrescreve a primeira. Uma pode ser chave de produção que não deveria estar ali.

| 11 | **Email server actions sem autenticação** — `sendWelcomeEmailAction(to, name)` pode ser chamado por qualquer client |

## 🔮 CONSIDERAÇÕES FUTURAS

**Pagamentos (Stripe):**
- Webhook endpoint em `/api/webhooks/stripe` com verificação de assinatura
- Nunca confiar em confirmação client-side
- Idempotency keys para criação de pagamento
- Chaves Stripe **nunca** em `NEXT_PUBLIC_`

**Vídeo:**
- Daily.co ou Twilio para video calls
- Tokens de sala com expiração curta
- Se área de saúde → considerar compliance HIPAA

**Upload de arquivos:**
- Validação de tipo/tamanho server-side
- Scan de malware (ClamAV ou serviço cloud)
- Supabase Storage com policies por bucket

**Mobile (futuro):**
- Certificate pinning
- Secure storage para tokens (Keychain/Keystore)
- Biometric auth
