# User Journey: Session Management

Last updated: 2026-03-29

## Goal

Enable users and professionals to manage scheduled sessions safely after booking.

## Actors

1. End user
2. Professional
3. Admin (for exception handling)
4. Cron workers (reminders and confirmation timeouts)

## Entry points

- `/agenda`
- Booking action components (confirm, cancel, reschedule, complete, no-show)
- Cron routes:
  - `/api/cron/booking-reminders`
  - `/api/cron/booking-timeouts`

## Happy path

1. Participants see upcoming and past sessions in agenda.
2. Professional confirms or updates session link when needed.
3. User can cancel/reschedule inside policy windows.
4. Professional can complete sessions after scheduled end.
5. Reminders are generated by cron job.
6. Expired manual confirmations auto-cancel and trigger refund status updates.

## Edge cases

1. Invalid state transitions are rejected by booking state machine guards.
2. No-show can be reported by either side under role constraints.
3. Reschedule requires notice window and conflict revalidation.

## Current implementation status

- `In progress` with strong baseline implemented.
- Cron reminder and timeout endpoints are live.
- State transition enforcement is implemented in server actions.

## Gaps

1. External delivery channels for reminders are limited.
2. Refund orchestration still requires full payment-provider integration.

## Next steps

1. Wire provider-level refund/capture behavior.
2. Expand observability for reminder delivery outcomes.
